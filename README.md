# runtime

The monorepo for `runtime.training` — a calendar-first website, a coach console, an organizer portal, an iOS app, and the FastAPI backend they all sit on.

Architecture decisions live in [`docs/architecture/001-stack-decisions.md`](docs/architecture/001-stack-decisions.md). Planning system in [`docs/planning/`](docs/planning/) — start at `_about.md`.

## Workspace layout

```
runtime/
├── apps/
│   ├── web/              Next.js — public site + /coach + /organizers
│   ├── api/              FastAPI (lands in Block 6)
│   └── ios/              SwiftUI (lands in Block 5)
├── packages/
│   ├── design-tokens/    Color, spacing, type, motion — single source of truth
│   ├── email/            React Email templates (lands when needed)
│   └── api-types/        OpenAPI-generated TS types (lands when needed)
├── supabase/
│   ├── migrations/       SQL migrations (run with `supabase db push`)
│   └── seed.sql          Seed data for local dev
└── docs/                 Planning, architecture, product design
```

`apps/api/`, `apps/ios/`, `packages/email/`, and `packages/api-types/` aren't created yet — they land with the blocks that need them.

## Running locally

Requires Node `>=20.11` (see `.nvmrc`). From the repo root:

```sh
npm install          # workspaces resolve packages/design-tokens automatically
npm run dev          # apps/web on http://localhost:3000
```

Other common commands, all from the root:

```sh
npm run build              # apps/web production build
npm run typecheck          # tsc --noEmit in apps/web
npm run lint               # ESLint in apps/web
npm run format             # prettier --write across the repo
npm run generate:tokens    # regenerate dist/ in packages/design-tokens
```

## Supabase (manual setup)

The `supabase/` directory holds migrations and seed data. The Supabase *project* (Postgres + Auth + Storage) is created once via the Supabase web UI:

1. Sign in at [supabase.com](https://supabase.com), create a project, region `eu-central-1`.
2. Copy the Project URL and `anon` key into `apps/web/.env.local`:
   ```
   NEXT_PUBLIC_SUPABASE_URL=...
   NEXT_PUBLIC_SUPABASE_ANON_KEY=...
   ```
3. Install the Supabase CLI (`brew install supabase/tap/supabase`) and link the project (`supabase link`).
4. `supabase db push` applies anything in `supabase/migrations/` to the cloud project. `supabase db reset` resets a local instance.

Nothing in `apps/web/` reads Supabase yet; this is set up so the next DB-touching pitch has somewhere to land.
