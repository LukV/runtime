# ship-pitch

Move a pitch out of in-cycle. Two paths: **ship** (it landed) and **drop** (overflowed appetite or no longer worth doing). The shipping path also releases via commitizen.

## Paths

- Pitch notes (active): `docs/planning/pitches/<name>.md`
- Shipped pitches: `docs/planning/pitches/shipped/<name>.md` (index at `shipped/_index.md`)
- In Cycle: `docs/planning/In Cycle.md`
- Repo changelog: `CHANGELOG.md` (auto-generated by `cz bump`)
- Released narrative: `docs/planning/Released.md`

## Steps for shipping

1. **Read the in-cycle pitch.** From `docs/planning/In Cycle.md`, confirm which pitch is currently in-cycle. Read the pitch note.

2. **Prompt for "what actually happened".** Ask Luk for a one-paragraph summary: what landed, what was cut, what surprised us, what to watch. Do not write it. This is Luk's reflection; the agent's job is to capture it cleanly.

3. **Update the pitch frontmatter.** Set `status: shipped`. Add `shipped_on: YYYY-MM-DD` (today). Keep all other fields.

4. **Write the "What actually happened" section.** Replace the placeholder block at the bottom of the pitch (`*(Fill in when the pitch ships...)*`) with Luk's paragraph from step 2.

5. **Move the pitch.** Move `docs/planning/pitches/<name>.md` to `docs/planning/pitches/shipped/<name>.md`. The `shipped/` directory and its `_index.md` already exist.

6. **Clear `In Cycle.md`.** Replace the `## Currently in cycle` section body with:

   ```
   ## Currently in cycle

   *Nothing in cycle yet.*

   See the on-deck list below for the next pickup.
   ```

7. **Release via commitizen.** Run `cz bump` in the repo root. This bumps the version per conventional-commits, updates `CHANGELOG.md`, and tags. Conventional commits should already have been authored during the cycle via `cz commit`.

   Commitizen config lives in `.cz.toml` at the repo root (not `pyproject.toml`, since this is a polyglot monorepo).

   If `cz bump` reports no version bump needed (e.g. only `chore:` commits), report this to Luk and ask whether to skip the bump or force one with `cz bump --increment PATCH`.

8. **Suggest the next pitch.** Read the on-deck list in `In Cycle.md` (or the relevant block note's pitches) and propose the next item. Do not auto-`start-pitch` — let Luk confirm.

9. **Released.md.** If shipping a pitch crosses a release boundary worth narrating (a block finishing, a milestone), prompt Luk to add a short entry to `docs/planning/Released.md`. Otherwise leave it — the changelog is enough for routine ships.

## Steps for dropping

When a pitch is dropped instead of shipped (overflowed appetite, no longer worth doing):

1. Set `status: dropped`. Add `ended: YYYY-MM-DD`.
2. Write "What actually happened" with explicit emphasis on **why dropped** — overflowed appetite, design changed under it, dependency slipped, no longer valuable.
3. **Do not move the file.** Dropped pitches stay in `docs/planning/pitches/` with `status: dropped` so the dead pitches are part of the record ([[../../../docs/planning/_about#Pitch lifecycle]]). Future agents reading the planning system should see what was attempted and abandoned.
4. Clear `In Cycle.md` as in step 6 above.
5. **Skip `cz bump`.** Nothing shipped.
6. Suggest the next pitch from on-deck.

## Don't

- Don't write the "what actually happened" yourself. The reflection is the value of the section; if you write it, the value evaporates.
- Don't bump version on a drop.
- Don't auto-start the next pitch.
- Don't delete dropped pitches — they're part of the planning record.
- Don't push the release tag. `cz bump` creates the tag locally; pushing is Luk's call (`git push --follow-tags`).
