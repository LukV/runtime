# Block 5 — iOS App, Downsized

The runner-facing product. Downsized per [[../../product-design/003-pilot-scope]] and per the strategic call to operate the coach as Wizard-of-Oz behind the scenes — the app shows a polished surface, but the intelligence behind it is An, the dietist. The runner doesn't know, doesn't need to know, and the moment we replace An with an LLM the surface stays unchanged.

**What "downsized" means.** Four screens that ship, three that don't (yet):

| Ships in v1 | Defers to v2 |
|---|---|
| Onboarding flow (8 screens) | In-app calendar |
| Vandaag (light + dark) | Wedstrijd sheet (use website) |
| Coach tab (messaging with An) | Week view |
| Settings sheet (minimal) | Race-day briefing |
|  | Post-race level-up flow |

The deferred surfaces are reachable from the website — runners can browse the calendar at runtime.training, including their own goal race. Wedstrijd in-app comes when WoZ stops scaling.

This is the **largest block** in the plan and the long pole. Started week 3, polished through week 9. Native Swift / SwiftUI. iOS-only — Android comes after the pilot validates the thesis.

App talks to the FastAPI backend from [[06-coach-backend-woz]] via device-token auth (no Supabase Auth in the app per [[../../architecture/001-stack-decisions#iOS auth: per-device tokens, not Supabase Auth]]).

Design: [[../../product-design/004-design-system-and-screens#3. The screens]] (every screen needed for v1 is already designed, except the settings sheet).

## Pitches

### Xcode project setup + design tokens import — *an evening*

New iOS 17+ SwiftUI project. Bundle ID `training.runtime.app`. Tokens from [[01-brand-system-in-code]] imported as a `Tokens.swift` enum. Asset catalog with the app icons from [[01-brand-system-in-code]]. Inter and JetBrains Mono bundled, registered in `Info.plist`. Source Serif either bundled or inlined-as-SVG per the open question in 01.

API client setup: `APIClient.swift` with the FastAPI base URL, device-token interceptor, basic retry. Generated Pydantic-typed models can be ported by hand or via OpenAPI codegen (`openapi-generator` or `swift-openapi-generator`).

### Claim code flow — *an evening*

The first-launch flow. App opens to a *"Welkom bij Runtime"* screen with a *"Voer je code in"* input. Runner types the 8-char code from their welcome email. App POSTs to `/api/auth/claim`, gets back a device token, stores it in the Keychain.

Subsequent launches: app reads token from Keychain, calls `/api/auth/whoami`, navigates straight to Vandaag.

*Failure modes:*
- Invalid/expired code → error message + *"Krijg een nieuwe via Luk (mailto)"*
- No internet → cached state, retry on next foreground
- Token invalidated → kick back to claim screen

### The onboarding flow — *a week of evenings*

Eight screens per [[../../product-design/004-design-system-and-screens#Onboarding — welcome|the onboarding design]]: welcome, three intake questions, goal, race-attach (or "geen wedstrijd, mijn doel volstaat"), plan preview, commitment moment.

All native SwiftUI. State carried in a single `OnboardingViewModel`. On the commitment screen, the runner data and chosen plan are POSTed to FastAPI (`POST /api/runner/onboarding/complete`), which creates the initial `plan` and seeds the first week of `workouts`. A confirmation triggers a welcome message in the Coach tab.

*Risk.* Plan preview screen — we're claiming a plan exists at this moment, but during WoZ the plan is *actually* generated by An manually. Solution: the preview screen shows a generic "weeks 1-4 opbouw / 5-9 bouwen / 10-11 pieken / 12 taper" block-level summary that's true of every plan, then the *actual* per-week structure arrives in the Coach tab from An over the first few days. This matches expectations from any commercial training app.

### Vandaag — light and dark — *a week of evenings*

The daily driver per [[../../product-design/004-design-system-and-screens#Vandaag — light mode]] and [[../../product-design/004-design-system-and-screens#Vandaag — dark mode]]. Real data: ribbon based on the runner's plan from FastAPI, headline + opbouw based on today's workout, adaptation row when (and only when) An adapted the plan that morning.

Light/dark switches with system setting. The dark-mode primary CTA is veldgroen per the design.

Data flow: app calls `GET /api/runner/today` on launch and pull-to-refresh. Response includes today's workout, ribbon state, and the current pending adaptation if any. UI renders from a single `TodayViewModel`.

### Coach tab — *a week of evenings*

Per [[../../product-design/004-design-system-and-screens#Coach conversation]]. Three states: empty (*An kijkt mee*), first message from An, mid-conversation. Both directions.

Messages from runner POST to `/api/runner/messages` (FastAPI fires a Slack webhook to An). Messages from An come from the coach console, polled by the app every 30 seconds when the tab is active and once per minute otherwise. Push notification on new message via APNs.

The runner never sees that this isn't AI. From a design standpoint, the surface is identical to how it'll work once a real LLM is wired up — only the backend swaps.

### Settings sheet — *an evening*

Minimal v1: change email, sign out, "geef feedback" (mailto Luk), "support" (mailto An), language picker (NL-BE only initially), and a small *"Over Runtime"* link to the website's *Over ons* page. No notification settings (everything default-on for the pilot). Gear icon top-right on Vandaag opens it as a sheet.

### Strava deep linking — *an evening*

Most pilot runners use Strava already. Two integrations:
1. **"Start de training" → Strava** — open the Strava app with the workout pre-loaded as a planned activity (Strava Routes API or just deep link to Strava's record screen with a known activity name). Optional, behind a toggle in settings.
2. **Read activity back** — An sees the runner's Strava activity in her console. V1: runners share their profile with An manually; V2: Strava API integration in FastAPI.

V1 ships option 1 only.

### Plan adaptation surfacing — *an evening*

The three adaptation screens designed in [[../../product-design/004-design-system-and-screens#Adaptation states — three triggers]]. Each fires as a modal *before* Vandaag the morning of adaptation.

`GET /api/runner/today` includes a `pending_adaptation` field when applicable. App shows the modal once on the next foreground, runner taps *"Akkoord"*, app POSTs to `/api/runner/adaptations/{id}/acknowledge`, modal dismisses to the adapted Vandaag.

*Out of scope for v1.* Weather adaptation as automatic — weather adaptations during WoZ are manually triggered by An after looking at a weather app. Real automation comes in v2 when we wire a weather API into FastAPI.

### Push notifications + APNs setup — *an evening*

Apple Developer Account ($99/yr) + APNs certificate, registered in the FastAPI deployment. App requests notification permission on first onboarding completion (not before — the runner has to commit before being asked for push perms).

Triggers handled by FastAPI per [[06-coach-backend-woz#Push notifications to runners]]:
1. New message from An
2. New adaptation fired
3. Optional: morning workout reminder (toggleable)

### TestFlight + App Store readiness — *a week of evenings, end of cycle*

App Store Connect account setup, screenshots (a clean Vandaag, the welcome, the commitment moment, the coach conversation, the dark mode Vandaag), App Store description (short, brand-voice, Dutch), privacy policy linked. TestFlight build for the 20 pilot runners *before* App Store release. Public release at the end of the two months.

*Risk.* App Store review timing. Apple sometimes takes a week. Build TestFlight first (no review needed) — pilot runners get the app via TestFlight invite, even before App Store approval lands.
